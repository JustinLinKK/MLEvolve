# MLEvolve Evidence Graph Schema

This folder now treats `graph_schema.yaml` as the canonical graph design for
MLEvolve hardware-aware memory.

The graph is an evidence store, not the scheduler control plane. SQLite/current
scheduler state remains authoritative for live jobs, commands, checkpoints,
events, and cache metadata. Neo4j stores measured profiling truth that future
agents can reuse.

## Canonical Model

Use these graph nodes:

- `(:Job:SingleJob)` for one model running alone, including training, runtime
  probes, memory probes, and batch-size probes.
- `(:Job:PackedJob)` for a concurrent packed/co-run execution group.
- `PackedJobMember` for per-model evidence inside a packed group.
- `Hardware`, `Model`, `TrainingConfig`, and `Technology` as reusable
  dimensions.

Use these relationships:

- `SINGLE_TRAINS_MODEL`
- `SINGLE_USES_CONFIG`
- `JOB_USED_HARDWARE`
- `JOB_USES_TECHNOLOGY`
- `HAS_PACKED_MEMBER`
- `MEMBER_TRAINS_MODEL`
- `MEMBER_USES_CONFIG`
- `MEMBER_USES_TECHNOLOGY`
- `MEMBER_BASELINE_SINGLE_JOB`

Do not add graph nodes for `Command`, `Event`, `Checkpoint`, `CacheEntry`, or
raw scheduler job payload mirrors. Those are operational records and belong in
the runtime store or log store.

## Data Boundary

Graph DB stores what happened:

- status, purpose, run scope, timestamps
- model/config/hardware/technology keys
- batch-size decisions
- runtime, throughput, utilization, VRAM/RAM measurements
- validation metrics and failure reasons
- compact `metrics_json`, `error_message`, and `notes` fallbacks

Vector DB stores how to change code:

- framework API documentation
- optimization recipes
- hardware-aware coding guidance
- source metadata and examples

The two stores connect through stable ontology keys such as `technology_key`,
`hardware_feature_key`, `model_family`, `workload_type`, `profile_symptom`,
`optimization_target`, and `api_symbol`.

## Legacy Files

`property_graph_schema.yaml` is retained only as legacy migration context for
the previous control-plane-plus-GraphRAG design. New code and docs should not
use its `RunProfile`, `RuntimeProfile`, `BatchProbeProfile`, `SoloProfile`, or
`PacketProfile` nodes as Neo4j labels.

## Migration Shape

Use a clean rebuild workflow:

1. Read authoritative runtime/profile records from SQLite and scheduler state.
2. Convert legacy profile concepts into canonical evidence:
   - batch probes and batch observations -> `SingleJob`
   - runtime profiles -> `SingleJob`
   - solo profiles -> `SingleJob`
   - pair/combination profiles -> `PackedJob` and `PackedJobMember`
   - terminal scheduler jobs -> `SingleJob`
3. Upsert dimensions and evidence nodes into Neo4j.
4. Keep rebuild dry-run by default; destructive graph wipes require an explicit
   operator flag.

Human-readable summaries should be generated by MCP responses, not persisted as
graph facts.
